Inkitt OS

The operating system I would build as Chief of Staff to replace Ali as the human router, so he only exercises judgment on things that genuinely require it.

Inkitt OS System Design Principles
Lossy at each level
Information compresses upward, never expands.
Cost-effective
Local processing over API calls.
Familiar tools
Slack + Notion. No new apps for ICs.
No platform jumping
ICs never leave Slack.
Proactive + Reactive
Morning lists + instant post-meeting pings.
See Logic
Lossy at each level: ICs see all their tasks. Managers see team patterns. Ali sees red flags only.

Cost-effective: Build 2 cuts Claude API to a flat rate regardless of meeting volume.

Familiar tools: Everyone already uses Slack and Notion. Adoption by default.

No platform jumping: Managers get dossiers in Slack. Ali gets his roll-up in Slack or his Office OS.

Proactive + Reactive: Morning to-do lists, end-of-day dossiers, pre-briefs. Post-meeting action pings immediately, blocker escalations on SLA breach.

See Logic
Compounding errors: AI agents can misclassify or mis-assign. The CoS sits between the agent and Ali as the quality gate. Nothing reaches Ali that the CoS has not verified.

Alert fatigue: Each layer filters. ICs see tasks. Managers see patterns. Ali sees red flags. The system reduces noise at every level.

Gaming: If someone marks everything ✅ without doing it, Org Health detects the pattern: high completion rate but flat OKR progress. Cross-referencing multiple signals catches anomalies.

Emoji adoption: Three reactions (✅ ❌ ❗) require zero learning. The CoS demos the full system in Week 4 with real examples from the discovery phase.
90-Day Rollout (12 weeks)
Weeks 1–3: Organizational Map
Build the organizational map. Attend every standup across all 6 teams. Understand the real information flow — where decisions actually get made, where handoffs break, where redundancies live. Document gaps between how teams think information moves and how it actually moves. This is the foundation the entire system is designed against.
Weeks 4–6: Pilot with 1–2 Teams
Notion Agent goes live with 1–2 teams. Validate tagging accuracy, test post-meeting pings and emoji reactions, tune the instruction page encoding CoS logic. Data starts accumulating. First manager dossier drafts generated. CoS reviews every output manually — no automation reaches Ali yet.
Weeks 7–9: Tune on Breakage
System has been running long enough to break. Tune based on real failure patterns: misclassified tasks, missed escalations, emoji adoption gaps, edge cases in cross-team dependencies. Refine the Notion Agent's instruction page. First Org Health Report generated from 6+ weeks of accumulated data. CoS stress-tests every layer before scaling.
Weeks 10–12: Scale to Full Org
Roll the revised OS out to all 6 teams. Emoji reactions live org-wide. Daily dossiers flowing to every manager. Org Health Reports running weekly for team leads and monthly for Ali. CoS operating at the audit + strategy layer: verifying escalations, finalizing CEO's Office OS, using Org Health insights to drive zero-to-one initiatives.
See Logic
Why 3 weeks of discovery before any automation? The CoS needs to understand the real information topology — not the org chart version, but how decisions actually flow. Without this, the Notion Agent's instruction page will encode assumptions instead of reality.

Why pilot with 1-2 teams? Small blast radius. If the agent misclassifies, it affects one team, not six. The CoS can manually review every output and tune before scaling.

Why a dedicated tuning phase? Every system breaks differently at scale. Weeks 7-9 are specifically reserved for fixing what broke during the pilot — not adding features, but hardening the foundation.

As Inkitt scales from 150 to 400 people, Build 1's costs grow linearly. Build 2's costs stay flat.
