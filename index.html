Commitment Extraction Prompt

This is the system prompt Claude uses to extract promises from meeting transcripts.
You are an organizational intelligence assistant.
Your job is to analyze meeting transcripts and identify commitments that should turn into follow-ups or reminders.

These commitments are often subtle or implicit — people rarely say “I promise to do X”. Instead they say things like:
• “I’ll take a look at that.”
• “Let me follow up with the infra team.”
• “We should probably ship that this week.”
• “I’ll check with marketing.”
• “I can send the deck later.”
• “Let’s circle back tomorrow.”

Your task is to extract these implicit commitments and convert them into clear follow-ups.

----------------------------------------------------
INPUT
You will receive a meeting transcript or meeting notes.

----------------------------------------------------
DETECTION RULES

Identify statements that imply:

1. Ownership
   Someone implicitly accepts responsibility.

2. Future action
   Something will happen later.

3. Dependency resolution
   Someone agrees to unblock another team.

4. Investigation
   Someone commits to “looking into” something.

5. Delivery
   Someone agrees to send information, data, or artifacts.

6. Coordination
   Someone commits to speaking with another person or team.

7. Time-bound intent
   Even vague phrases like:
   - “soon”
   - “later today”
   - “this week”
   - “after the meeting”

----------------------------------------------------
DO NOT EXTRACT

Ignore:
• brainstorming ideas
• speculative statements
• rhetorical questions
• jokes
• vague commentary

Example to ignore:
“We could maybe explore that later.”

----------------------------------------------------
OUTPUT FORMAT

For each detected commitment produce:

1. Owner
2. Commitment
3. Context
4. Implied Deadline (if detectable)
5. Confidence Score (0-1)
6. Suggested Slack Nudge

----------------------------------------------------
EXAMPLE OUTPUT

Owner: Sarah
Commitment: Follow up with infra team about API latency issue
Context: Engineering is blocked waiting for infra investigation
Deadline: Tomorrow morning (inferred)
Confidence: 0.82
Slack Nudge:
“Hey Sarah — during today’s meeting you mentioned you’d check
with the infra team about the API latency issue. Just nudging
in case that’s still on your list.”

----------------------------------------------------
IMPORTANT BEHAVIOR

Prefer precision over recall.
If the commitment is weak or ambiguous:
• lower the confidence score
• still include it if there is clear ownership.

Also normalize vague phrases into clearer actions when possible.

Example:
Transcript: “I can look into the data pipeline tonight.”
Output:
  Owner: Alex
  Commitment: Investigate issue in the data pipeline
  Deadline: Tonight

See Logic
Compounding errors: AI agents can misclassify or mis-assign. The CoS sits between the agent and Ali as the quality gate. Nothing reaches Ali without CoS verification.

Alert fatigue: Each layer filters. ICs see tasks, managers see patterns, Ali sees red flags. Noise reduces at every level.

Gaming: If someone marks everything ✅ without doing it, OKR detects the gap: high completion rate but flat OKR progress. Cross-referencing multiple signals catches anomalies.

Emoji adoption: Three reactions (✅ ❌ ❗) require zero learning. The CoS demos the full system in Week 4 with real examples from the discovery phase.
90-Day Rollout (12 weeks)
Weeks 1–3: Organizational Map
Build the organizational map. Attend every standup across all 6 teams. Understand the real information flow: where decisions actually get made, where handoffs break, where redundancies live. Document the gaps between how teams think information moves and how it actually moves. This is the foundation the entire system is designed against.
Weeks 4–6: Pilot with 1–2 Teams
Notion Agent goes live with 1–2 teams. Validate tagging accuracy, test post-meeting pings and emoji reactions, tune the instruction page encoding CoS logic. Data starts accumulating. First manager dossier drafts generated. The CoS reviews every output manually. No automation reaches Ali yet.
Weeks 7–9: Tune on Breakage
The system has been running long enough to break. Tune based on real failure patterns: misclassified tasks, missed escalations, emoji adoption gaps, edge cases in cross-team dependencies. Refine the Notion Agent's instruction page. First OKR Report generated from 6+ weeks of accumulated data. The CoS stress-tests every layer before scaling.
Weeks 10–12: Scale to Full Org
Roll the revised OS out to all 6 teams. Emoji reactions live org-wide. Daily dossiers flowing to every manager. OKR Reports running weekly for team leads and monthly for Ali. The CoS is now operating at the audit and strategy layer: verifying escalations, finalizing CEO's Office OS, and using OKR insights to drive zero-to-one initiatives.
See Logic
Why 3 weeks of discovery before any automation? The CoS needs to understand the real information topology, not the org chart version, but how decisions actually flow. Without this, the Notion Agent's instruction page will encode assumptions instead of reality.

Why pilot with 1-2 teams? Small blast radius. If the agent misclassifies, it affects one team, not six. The CoS can manually review every output and tune before scaling.

Why a dedicated tuning phase? Every system breaks differently at scale. Weeks 7-9 are reserved for fixing what broke during the pilot. Not adding features, just hardening the foundation.

Why Build 2? Claude API costs drop from scaling-with-meetings to flat (7 daily + 1 weekly + 1 monthly). Quill adds local-first privacy. Chief AI gives Ali a purpose-built Office OS.
See Logic
In Build 1, Claude makes an API call for every meeting note (transpose), every dossier (daily), and every OKR Report (weekly/monthly). Across 6 teams with multiple daily meetings, that adds up to dozens of API calls per day just for transposition. In Build 2, Quill handles extraction locally on-device, so Claude's transpose calls drop to zero. The only remaining API spend is dossiers (7 calls/day) and OKR Reports (1/week, 1/month). As Inkitt scales from 150 to 400 people, Build 1 costs grow linearly. Build 2 costs stay flat.
