Inkitt OS
The operating system I would build as Chief of Staff to replace Ali as the human router, so he only exercises judgment on things that genuinely require it.
System design principles
Lossy at each level
Every layer filters. ICs see all their tasks. Managers see team patterns. Ali sees red flags only. Information compresses upward, never expands.
Cost-effective performance
Minimize expensive API calls. Use local processing and native tool intelligence where possible. Build 2 cuts Claude API costs to a flat rate regardless of meeting volume.
Fewer, familiar tools
Everyone already uses Slack and Notion. I build on top of existing habits, not against them. No new apps for ICs to learn. Adoption by default.
Minimize platform jumping
ICs never leave Slack. Managers get dossiers in Slack. Ali gets his roll-up in Slack or his Office OS. Nobody opens Notion to update a task status.
Proactive and reactive
Proactive: morning to-do lists, end-of-day dossiers, pre-briefs before Ali's meetings. Reactive: post-meeting action pings immediately after every call, blocker escalations on SLA breach.

Why pursue Build 2?
In Build 1, Claude makes an API call for every meeting note (transpose), every dossier (daily), and every Org Health Report (weekly/monthly). Across 6 teams with multiple daily meetings, that is dozens of Claude API calls per day for transposition alone. In Build 2, Quill handles extraction locally on-device. Claude's transpose calls drop to zero. The only remaining API spend is dossiers (7 calls/day) and Org Health Reports (1/week, 1/month). As Inkitt scales from 150 to 400 people, Build 1's costs grow linearly. Build 2's costs stay flat. Additionally: Quill adds local-first privacy for sensitive CEO meetings, and Chief AI gives Ali a purpose-built Office OS instead of one I maintain manually.
